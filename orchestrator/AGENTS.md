# Orchestrator — AGENTS.md

Этот модуль — **будущий автоматический пайплайн** агентной разработки EqSiteCMS.

Текущее состояние мануального пайплайна: [`WORKFLOW.md`](../WORKFLOW.md).

---

## Что уже реализовано

### Модули

| Файл | Что делает |
|---|---|
| `main.py` | Точка входа. Парсит `--task-id` и `--prompt`, создаёт git-ветку, инициализирует пайплайн |
| `pipeline.py` | Оркестратор с топологической сортировкой. Управляет задачами, retry-логикой, генерирует отчёт |
| `executor.py` | Запускает Cursor CLI (`cursor -p <prompt> --model <model>`) для каждой задачи |
| `plan_parser.py` | Парсит `docs/plans/<TICKET>.md`, находит незавершённые `- [ ]` пункты по секциям агента |
| `task.py` | Датакласс `Task` со статусами: `PENDING / IN_PROGRESS / COMPLETED / FAILED` |
| `config.py` | Модели агентов, лимиты итераций, пути директорий |

### Текущий пайплайн (`make task NEX-XXX "описание"`)

```
Planner → (генерирует docs/plans/NEX-XXX.md)
    ↓
Backend ← (зависит от Planner)
Frontend ← (зависит от Planner, параллельно с Backend)
    ↓
Quality Gate ← (зависит от Backend + Frontend)
```

### Retry-механизм

После выполнения каждого агента `plan_parser` проверяет незакрытые `- [ ]` пункты.
Если найдены — создаётся retry-задача. Лимит: `MAX_ITERATIONS_PER_TASK_LINEAGE = 3`.

### Формат совместимости с plan_parser

Секции чеклиста должны называться строго:
- `### Backend`
- `### Frontend`
- `### Quality Gate`

Пункты: `- [ ] описание` → после выполнения: `- [x] описание`

## Роадмап автоматизации

### Фаза 1 — Human-in-the-loop (следующий шаг)

**Проблема:** сейчас оркестратор запускает всё автоматически без паузы для ревью плана.

**Нужно реализовать:**
- [ ] После Planner: пауза с выводом плана и запросом `[y/n] Продолжить?`
- [ ] Режим `--dry-run`: только генерирует план, не запускает агентов
- [ ] Режим `--from-plan NEX-XXX.md`: пропускает Planner, берёт готовый план
- [ ] Режим `--only Backend`: запускает только конкретного агента

### Фаза 2 — State persistence

**Проблема:** если оркестратор упал на середине — он начинает с начала.

**Нужно реализовать:**
- [ ] Сохранять состояние пайплайна в `docs/plans/NEX-XXX-state.json`
- [ ] `--resume NEX-XXX`: продолжить с последней точки
- [ ] Статусы сохранять между перезапусками

### Фаза 3 — Quality Gate как автоматический шаг

**Нужно реализовать:**
- [ ] QG запускает `make test` и `make lint` перед ревью diff
- [ ] Если тесты красные — автоматически возвращает на доработку без спроса
- [ ] QG сохраняет `docs/plans/NEX-XXX-review.md` со статусом `APPROVED` / `REWORK`
- [ ] При `REWORK` — rework-чеклист передаётся агенту как новая задача

### Фаза 4 — Параллельное выполнение

**Нужно реализовать:**
- [ ] Backend и Frontend запускаются параллельно (сейчас sequential)
- [ ] Использовать `asyncio` или `ThreadPoolExecutor` в `pipeline.py`

### Фаза 5 — Интеграция с трекером задач

**Нужно реализовать:**
- [ ] Читать задачи из Linear / GitHub Issues по ID
- [ ] Автоматически создавать PR после `APPROVED`
- [ ] Комментировать в Linear статус выполнения

### Фаза 6 — Мониторинг и веб-UI

- [ ] Простой веб-интерфейс для просмотра статуса пайплайна
- [ ] Логи агентов в реальном времени
- [ ] История запусков

## Запуск оркестратора

```bash
# Полный пайплайн: ветка + план + код + ревью
make task NEX-XXX "добавить сущность Job с CRUD API"

# Или напрямую
uv run python -m orchestrator.main --task-id NEX-XXX --prompt "добавить сущность Job"
```

### Переменные окружения

```bash
# .env в корне проекта
CURSOR_MODEL=auto          # модель по умолчанию для всех агентов
PLANNER_MODEL=auto         # переопределить для Planner
QUALITY_GATE_MODEL=auto    # переопределить для QG
```

## Архитектурные решения

### Почему Cursor CLI, а не LangChain/LlamaIndex?

Cursor CLI (`cursor -p`) даёт агентам полный доступ к кодовой базе (индекс, поиск, правки файлов)
без необходимости передавать контекст вручную. Агенты работают в реальном workspace.

### Почему plan_parser парсит markdown, а не JSON?

Планы должны быть читаемы и редактируемы человеком. Markdown + чеклисты — минимальный
overhead при максимальной совместимости с Cursor и GitHub.

### Почему не используем граф задач из фреймворков (LangGraph, CrewAI)?

Текущая реализация намеренно простая: топологическая сортировка + retry. Это позволяет
полностью контролировать поведение без зависимости от внешних фреймворков.
