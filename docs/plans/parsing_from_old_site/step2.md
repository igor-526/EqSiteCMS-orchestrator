## Контекст

Этап 1 собрал 136 страниц конкретных лошадей и пони старого Joomla/K2-сайта. Этап 2 должен согласовать инструкцию, по которой позднее можно будет собрать системный JSON по этим страницам.

На этом этапе парсер не реализуется. Документ фиксирует источники, правила поиска атрибутов, нормализацию значений, риски и чеклист согласования.

Проверенные реальные страницы:

| Тип | Страница | Наблюдение |
|---|---|---|
| horse | `/index.php/ferma/horses/item/202-gospoja` | простая карточка, галерея `sigPro`, пустое потомство |
| horse | `/index.php/ferma/horses/gan-trk/valter` | родословная с фото/ссылками, услуга случки, модуль потомства |
| pony | `/index.php/ferma/pony/appaluza-pony/florian` | длинное потомство, внешние pedigree-ссылки `allbreedpedigree.com` |
| pony | `/index.php/ferma/pony/shetlend/knopka` | простая родословная, статус `Продана`, одно потомство |

Дополнительно проверено:

- разметка K2 item находится в `#k2Container.itemView`;
- основной HTML контента находится в `.itemFullText`;
- исходный K2 `fulltext` хранит тот же контент с `{gallery}...{/gallery}`;
- на проверенных страницах `aj4tr_k2_items.extra_fields = []`, поэтому extra fields нельзя считать источником атрибутов;
- галереи в rendered HTML раскрываются в `ul.sigProContainer a.sigProLink[href]`;
- потомство выводится модулем `mod_bt_contentslider` как блоки `.bt-inner`.

## Цель

Согласовать инструкцию извлечения всех атрибутов для будущего JSON:

```json
{
  "path": "string",
  "name": "string",
  "description": "string | null",
  "services": ["string"],
  "sex": "male | female | null",
  "breed": "string | null",
  "coat_color": "string | null",
  "kind": "horse | pony",
  "bdate": "YYYY-MM-DD | YYYY | null",
  "height": "number | string | null",
  "photos": ["string"],
  "pedigree": {
    "sire": "PedigreeNode | null",
    "dam": "PedigreeNode | null",
    "children": ["ChildNode"]
  }
}
```

`PedigreeNode`:

```json
{
  "name": "string",
  "path": "string | null",
  "photos": ["string"],
  "owner": "string | null",
  "coat_color_short": "string | null",
  "bdate": "YYYY-MM-DD | YYYY | null",
  "pedigree": {
    "sire": "PedigreeNode | null",
    "dam": "PedigreeNode | null"
  }
}
```

`ChildNode`:

```json
{
  "name": "string",
  "path": "string | null",
  "photos": ["string"],
  "sex": "male | female | null",
  "breed": "string | null",
  "bdate": "YYYY-MM-DD | YYYY | null"
}
```

## Детали реализации

### Источники данных и приоритеты

| Атрибут | Основной источник | Резервный источник | Правило |
|---|---|---|---|
| `path` | `Path` из `step1.md` | canonical link текущей страницы | Хранить относительный путь без `http://localhost:8080` |
| `kind` | `Type` из `step1.md` | URL содержит `/horses/` или `/pony/` | `horse` для лошадей, `pony` для пони |
| `name` | `meta[property="og:title"]` или `<title>` | очищенный `.itemTitle` | Предпочитать короткое имя без племенного номера |
| `breed` | таблица характеристик справа, label `Порода` | introtext/meta description | Сохранять текст как на сайте, нормализуя пробелы |
| `sex` | таблица характеристик, label `Пол` | introtext | Маппинг: `жеребец` -> `male`, `мерин` -> `male`, `кобыла` -> `female`; иначе `null` |
| `bdate` | таблица характеристик, label `Дата рождения` | introtext | `DD.MM.YYYY` -> `YYYY-MM-DD`, одиночный год -> `YYYY`, пусто -> `null` |
| `coat_color` | таблица характеристик, label `Масть` | introtext | Сохранять полное значение, нормализуя `темно`/`тёмно` только если будет согласован словарь |
| `height` | таблица характеристик, label `Высота в холке` | pedigree/introtext только при явном формате высоты | Пусто -> `null`; число с `см`/`m` сохранить после нормализации |
| `photos` | `ul.sigProContainer a.sigProLink[href]` | `{gallery}folder...{/gallery}` + файлы в `/images` | Сохранять оригинальные пути картинок, не cache/thumbnail |
| `description` | `.caption` блоки в правой колонке после характеристик | introtext после строки-разделителя | Исключить характеристики, услуги, родословную и потомство |
| `services` | `.caption` блоки с коммерческими/status label | introtext | Сохранять строки `Предлагается к случке`, `Продается`, `Продан`, `Продана` только если есть содержательное значение |
| `pedigree.sire` | первая ячейка первого поколения в таблице `Родословная` | plain text из `articleBody` | Считать `null`, если ячейка пустая |
| `pedigree.dam` | первая ячейка второго блока первого поколения в таблице `Родословная` | plain text из `articleBody` | Считать `null`, если ячейка пустая |
| `pedigree.children` | `.bt-cs .bt-inner` после заголовка `Потомство` | plain text после `Потомство` | Каждый `.bt-inner` = один потомок |

### Общая последовательность для каждой страницы

1. Взять очередную строку из `step1.md`:
   - `Type`;
   - `Title`;
   - `Path`;
   - `K2 item id`.
2. Загрузить rendered HTML по `http://localhost:8080{Path}`.
3. Проверить, что страница содержит `#k2Container.itemView` и `.itemFullText`.
4. Зафиксировать `path` строго из `step1.md`, а не из текущего redirect/canonical, чтобы не потерять выбранный canonical этапа 1.
5. Определить `kind` из `Type`.
6. Извлечь `name` из `og:title`/`<title>`. Если доступен только `.itemTitle`, очистить:
   - начальные номера вида `489 `;
   - хвостовые племенные номера вида ` 47`, только если `og:title` подтверждает короткое имя;
   - скобочное английское имя не удалять без отдельного решения, если оно часть короткого имени.
7. В `.itemFullText` найти первую таблицу карточки:
   - левая колонка обычно содержит `sigPro` галерею;
   - правая колонка содержит характеристики и описательные `.caption`.
8. Извлечь характеристики из правой колонки по label-ам, а не по позиции строки.
9. Извлечь галерею из `sigPro`.
10. Найти таблицу после заголовка `Родословная` и разобрать first-level родителей.
11. Найти блоки потомства в `.bt-cs .bt-inner` после заголовка `Потомство`.
12. Применить нормализацию, собрать JSON-объект.
13. Записать диагностические предупреждения для страницы, если обязательные для проверки зоны не найдены.

### Правила поиска характеристик

Правую колонку карточки нужно разбирать как набор label/value пар. В rendered HTML label обычно выделен через `<strong>`/`<em>`:

- `Порода`;
- `Пол`;
- `Дата рождения`;
- `Масть`;
- `Высота в холке`.

Требования к поиску:

- сравнивать label после удаления `NBSP`, лишних пробелов, двоеточий и HTML-тегов;
- не опираться на конкретные `style`, ширины таблиц или шрифты;
- значения брать из текста после двоеточия до следующего `<br>`/label;
- пустые значения после двоеточия превращать в `null`;
- при конфликте rendered HTML и `introtext` приоритет у rendered HTML.

### Нормализация значений

#### Текст

- заменить `&nbsp;`/Unicode NBSP на обычный пробел;
- схлопнуть повторяющиеся пробелы;
- убрать пустые строки и строки-разделители из `_`, `—`, `-`;
- декодировать HTML entities;
- не менять регистр пород, мастей и имен без отдельного словаря.

#### Даты

| Вход | Выход |
|---|---|
| `21.05.2007` | `2007-05-21` |
| `05.04.2000 г.р.` | `2000-04-05` |
| `1974` | `1974` |
| пусто | `null` |

Если дата неполная и не является одиночным годом, сохранить `null` и добавить warning.

#### Пол

| Вход | Выход |
|---|---|
| `жеребец` | `male` |
| `мерин` | `male` |
| `кобыла` | `female` |
| другое/пусто | `null` + warning |

### `description`

`description` собирается из правой колонки первой таблицы, из `.caption` блоков после характеристик.

Исключить из `description`:

- строки характеристик;
- строки услуг/status (`Предлагается к случке`, `Продается`, `Продан`, `Продана`);
- заголовки `Родословная`, `Потомство`;
- содержимое таблицы родословной;
- содержимое `btcontentslider`.

Если после очистки нет содержательного текста, ставить `null`.

Примеры:

- Вальтер: `Рожден в хозяйстве КЗ "Кировский"`;
- Флориан: текст про леопардовую масть, Голландию, паспорт и диплом;
- Госпожа: `null`.

### `services`

В `services` включать коммерческие или статусные строки, найденные в `.caption`/introtext:

- `Предлагается к случке: 15 000 руб.`;
- `Продается: 200 000 руб.`;
- `Продан: <место/комментарий>`;
- `Продана: <место/комментарий>`.

Правила:

- сохранить строку целиком после нормализации пробелов;
- если label есть, но значение пустое (`Продана:` без места/суммы), по умолчанию не добавлять в `services`, а записать warning;
- если страница содержит несколько строк услуг/status, сохранить все в порядке появления;
- не пытаться классифицировать услугу в отдельные поля до согласования будущей модели.

Открытый вопрос: нужно ли хранить пустой статус `Продана:` как `services: ["Продана:"]` или игнорировать. Рекомендуемое правило для этапа 2: игнорировать пустые статусы, чтобы JSON не содержал слабые факты.

### `photos`

Основное правило:

- брать только `href` у `a.sigProLink` внутри `ul.sigProContainer`;
- не брать `img[src="/plugins/.../transparent.gif"]`;
- не брать `background-image` из `/cache/jw_sigpro/...`;
- сохранять порядок галереи;
- дедуплицировать одинаковые пути.

Если rendered HTML недоступен или `sigPro` не раскрылся, fallback:

1. Найти в исходном `fulltext` токен `{gallery}home/horse/valter:450:350:1:{/gallery}`.
2. Преобразовать folder в `/images/home/horse/valter`.
3. Прочитать файлы из `/home/igor/projects/ad_joomla/src/images/home/horse/valter`.
4. Исключить `index.html` и не-image файлы.
5. Сформировать пути `/images/.../<file>`.

Не использовать `media/k2/items/cache/*` как основное фото, потому что это K2 cache/thumbnail, а не исходник галереи.

### `pedigree.sire` и `pedigree.dam`

Родословная обычно представлена одной таблицей после заголовка `Родословная`:

- первый столбец, первая ячейка с `rowspan="4"` = отец (`sire`);
- первый столбец, следующая ячейка с `rowspan="4"` после первых четырех строк = мать (`dam`);
- второй и третий столбцы дают второй/третий уровень pedigree.

Правила для `PedigreeNode`:

- `name`: предпочитать текст первого `<a>` в ячейке; если ссылки нет, взять первый значимый текст до первой запятой/перевода строки;
- `path`: `href` первого `<a>`, если есть; для внешних ссылок хранить без локальной подстановки, протокол можно нормализовать отдельно при импорте;
- `photos`: все `img[src]` внутри ячейки, кроме пустых/служебных;
- `owner`: текстовая строка между именем и строкой масти/года, если она явно отделена `<br>` или запятой;
- `coat_color_short`: короткая масть из фрагментов вида `гн.`, `вор.`, `рыж.`, `т.-гн.`, `бул.`, `SPOTTED`;
- `bdate`: год или дата из ячейки;
- `pedigree`: вложенные `sire`/`dam` второго уровня, если соседние ячейки заполнены.

Цвет ячейки можно использовать только как вспомогательный сигнал:

- `#cbf5a9` часто соответствует мужской линии;
- `#f5e29f` часто соответствует женской линии.

Не делать цвет единственным источником, потому что часть ячеек не содержит class/цвет или содержит произвольную верстку.

Если ячейка содержит только `&nbsp;`/пустой текст, соответствующий узел должен быть `null`.

### `pedigree.children`

Потомство чаще всего отрисовано модулем `mod_bt_contentslider`:

```text
.bt-cs .bt-inner
  a.bt-image-link[href][title]
  img.hovereffect[src][alt]
  a.bt-title[href]
  .bt-introtext
```

Правила:

- каждый `.bt-inner` = один `ChildNode`;
- `name`: `a.bt-title` text, fallback `a.bt-image-link[title]`, fallback `img[alt]`;
- `path`: `href` у `a.bt-title`, fallback `a.bt-image-link`;
- `photos`: `img.hovereffect[src]`; если это cache URL, оставить как найдено для диагностического JSON, но пометить как cache-derived warning;
- `breed`: первая строка `.bt-introtext` до строки пола/даты;
- `sex`: из строки вида `жеребец, 27.02.2015 г.р.` или `кобыла, 04.03.2017 г.р.`;
- `bdate`: дата из той же строки;
- родительские ссылки внутри `.bt-introtext` не включать в `ChildNode`, потому что они дублируют связь потомка с текущей страницей и вторым родителем.

Если после `Потомство` нет `.bt-inner`, `children` = `[]`.

### Валидация результата

Для каждой страницы будущий парсер должен сформировать не только JSON, но и технический отчет качества:

| Проверка | Ожидание |
|---|---|
| `path` | совпадает с `step1.md` |
| `kind` | совпадает с `Type` из `step1.md` |
| `name` | не пустой |
| `photos` | допускается пустой массив, но warning если нет ни `sigPro`, ни `{gallery}` |
| `breed/sex/bdate/coat_color` | warning при отсутствии, не hard fail |
| `pedigree` | объект всегда есть; `sire`/`dam` могут быть `null` |
| `children` | массив всегда есть |
| даты | ISO `YYYY-MM-DD` или `YYYY`; иначе `null` + warning |
| ссылки | локальные ссылки относительные, внешние ссылки сохранены как внешние |

Минимальная ручная выборочная проверка после реализации будущего парсера:

- 5 страниц horse из разных категорий;
- 5 страниц pony из разных категорий;
- страницы с `children = []`;
- страницы с 1 ребенком;
- страницы с 5+ детьми;
- страницы с внешними pedigree-ссылками;
- страницы с пустой/частичной родословной;
- страницы с `services`.

## Открытые вопросы и риски

1. `name`: нужно ли удалять английское имя в скобках (`Флориан (Florian van de Kruisstraat)`) или хранить как часть `name`.
2. `services`: нужно ли хранить пустые статусы `Продана:`/`Продан:` без значения.
3. `height`: неизвестно, встречаются ли значения в сантиметрах и метрах одновременно; до реализации нужно проверить все 136 страниц.
4. `coat_color`: нужна ли словарная нормализация (`темно-гнедая` -> `тёмно-гнедая`) или важно сохранить текст старого сайта.
5. `pedigree.path`: нужно ли хранить внешние URL с протоколом или нормализовать до host/path как в примере.
6. `children.photos`: модуль потомства отдает cache-картинки, не всегда исходные фото; нужно решить, достаточно ли cache URL для потомков или нужно дополнительно резолвить страницу ребенка.
7. Потомки в слайдере могут быть страницами, которые уже есть в `step1.md`, но их `path` может отличаться от canonical из этапа 1. Рекомендуется позже делать cross-reference по K2 item id или normalized path.
