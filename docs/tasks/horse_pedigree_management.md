# Контекст
В плане `docs/plans/feature/horses_management` мы реализовали раздел управления лошадьми. Там же мы реализовали модальное окно с заглушкой под управление родословной лошади. У нас есть для управления родословной следующие эндпоинты
- **POST** `/api/horses/{horse_id}/pedigree` для установки связей
- **GET** `/api/horses/{horse_id}/pedigree/{mode}` для поиска доступных лошадей для связи
Нам нужно реализовать фронт для управления этой родословной и проверить корректность работы валидации
# Backend
Оба эндпоинта должны работать на одних принципах валидации. Один должен отдавать список, подходящий под условия валидации, а второй проверять соответствие предложенной лошади для установки связи
Там уже есть валидация, но, вероятно, учтено не всё. Нужно проверить валидацию на основе нижеизложенных логических принципов
## Логика валидации
### Валидация выбора отца

- `sex = male`
- `id != currentHorse.id`
- `id != pedigree.dam.id`
- `id` отсутствует среди `pedigree.foals[].id`
- `bdate < currentHorse.bdate`
### Валидация выбора матери
- `sex = female`
- `id != currentHorse.id`
- `id != pedigree.sire.id`
- `id` отсутствует среди `pedigree.foals[].id`
- `bdate < currentHorse.bdate`
### Валидация выбора потомка
- `id != currentHorse.id`
- `id != pedigree.sire.id`
- `id != pedigree.dam.id`
- `id` отсутствует среди `pedigree.foals[].id`
- `bdate > currentHorse.bdate`

#  UX/UI модального окна управления родословной
## 1. Назначение
Модальное окно предназначено для просмотра и редактирования ближайшей родословной выбранной лошади:
- сама лошадь;
- отец;
- мать;
- потомство;
- действия над связями: удалить, заменить, перейти.
Окно не должно отображать полное дерево поколений. В нём показывается только один уровень родителей и один уровень потомков.
---
## 2. Источник данных
Основной объект — DTO лошади.
### Данные текущей лошади брать из:
- `id`
- `slug`
- `name`
- `sex`
- `kind`
- `bdate`
- `bdate_formatted`
- `bdate_mode`
- `coat_color.name`
- `coat_color.short_name`
- `breed.name`
- `breed.short_name`
- `photos`
- `pedigree`
### Данные отца брать из:
- `pedigree.sire`
### Данные матери брать из:
- `pedigree.dam`
### Данные потомства брать из:
- `pedigree.foals`
---
## 3. Общая структура модального окна
Модальное окно состоит из четырёх основных зон:
1. Шапка модального окна.
2. Блок родителей.
3. Блок потомства.
4. Нижняя панель действий.
Расположение сверху вниз:
```
[ Шапка модального окна ][ Родители ]Отец  ----  Текущая лошадь  ----  Мать[ Потомство ]Карточка потомка | Карточка потомка | Карточка потомка[ Нижняя панель действий ]
```
---
## 4. Шапка модального окна

В шапке отображается:
```
Родословная — {name} {sex icon}
```

Где:
- `{name}` брать из `name`.
- Иконку пола брать из `sex`.
### Правила отображения пола
- `sex = male` → мужской символ.
- `sex = female` → женский символ.
- `sex = geld` → можно отображать отдельную метку “мерин” или нейтральный вариант, если в системе нет отдельной иконки.
- если `sex` отсутствует или неизвестен → иконку не показывать.
Справа в шапке должна быть кнопка закрытия модального окна.
---
## 5. Блок “Родители”
Блок начинается с заголовка:
```
Родители
```
Внутри блока расположены три карточки в одну горизонтальную линию:
```
[ Отец ] —— [ Текущая лошадь ] —— [ Мать ]
```
### 5.1. Карточка текущей лошади
Карточка текущей лошади располагается по центру.

В карточке отображаются:
1. Фото или плейсхолдер.
2. Кличка.
3. Иконка пола.
4. Пол + год рождения.
5. Масть.

Данные:
- кличка: `name`
- пол: `sex`
- дата рождения: `bdate_formatted` или `bdate`
- масть: `coat_color.name` или `coat_color.short_name`
- фото: главное фото из `photos`

#### Правила выбора фото
Если в `photos` есть фото с `is_main = true`, использовать его.
Если главного фото нет, но массив `photos` не пустой, использовать первое фото из массива.
Если `photos` пустой, показывать плейсхолдер без фотографии.
#### Правила отображения даты рождения
Предпочтительно использовать `bdate_formatted`.
Если `bdate_formatted` отсутствует:
- если `bdate_mode = y`, показывать только год из `bdate`;
- если `bdate_mode = ym`, показывать месяц и год;
- если `bdate_mode = ymd`, показывать полную дату;
- если `bdate_mode = hide`, дату не показывать.
В карточке текущей лошади не должно быть меню действий.

---
### 5.2. Карточка отца
Над карточкой отца, вне самой карточки, отображается подпись:
```
Отец
```
Внутри карточки отображаются данные из `pedigree.sire`.
Если `pedigree.sire` заполнен, показывать:
1. Фото или плейсхолдер.
2. Кличку.
3. Иконку пола.
4. Пол + год рождения.
5. Масть.
6. Меню дополнительных действий в правой части карточки.
Данные:
- кличка: `pedigree.sire.name`
- пол: `pedigree.sire.sex`
- дата рождения: `pedigree.sire.bdate_formatted`
- масть: `pedigree.sire.coat_color.name` или `pedigree.sire.coat_color.short_name`
- фото: `pedigree.sire.photos`
#### Если отец отсутствует
Если `pedigree.sire = null`, вместо заполненной карточки показывать карточку отсутствующего производителя.
Внутри карточки:
```
Не указан——
```
Над карточкой всё равно остаётся подпись:
```
Отец
```
В карточке отсутствующего отца также должно быть меню дополнительных действий.
Меню для отсутствующего отца должно содержать только релевантные пункты:
- Заменить связь
Пункт “Удалить связь” не нужен, потому что связи нет.
Пункт “Перейти” не нужен, потому что некуда переходить.

---
### 5.3. Карточка матери
Над карточкой матери, вне самой карточки, отображается подпись:
```
Мать
```
Внутри карточки отображаются данные из `pedigree.dam`.
Если `pedigree.dam` заполнена, показывать:
1. Фото или плейсхолдер.
2. Кличку.
3. Иконку пола.
4. Пол + год рождения.
5. Масть.
6. Меню дополнительных действий в правой части карточки.
Данные:
- кличка: `pedigree.dam.name`
- пол: `pedigree.dam.sex`
- дата рождения: `pedigree.dam.bdate_formatted`
- масть: `pedigree.dam.coat_color.name` или `pedigree.dam.coat_color.short_name`
- фото: `pedigree.dam.photos`
#### Если мать отсутствует
Если `pedigree.dam = null`, показывать карточку отсутствующего производителя.
Внутри карточки:
```
Не указана——
```
Меню для отсутствующей матери:
- Заменить связь
---
## 6. Линии связи между карточками
В блоке родителей должна визуально отображаться связь:
```
Отец —— Текущая лошадь —— Мать
```
Линия отца соединяет карточку отца с карточкой текущей лошади.
Линия матери соединяет карточку матери с карточкой текущей лошади.
Если отец или мать отсутствует, линия всё равно может отображаться до карточки “Не указан / Не указана”, чтобы пользователь понимал, где находится слот родителя.

---
## 7. Блок “Потомство”

Блок располагается ниже блока родителей и отделён от него отдельной секцией.
В верхней части блока:
слева:
```
Потомство
```
справа:
```
+ Добавить потомка
```
Кнопка “Добавить потомка” должна быть только одна — в заголовочной зоне блока потомства.
Отдельной карточки “Добавить потомка” внутри списка быть не должно.

---
## 8. Список потомства
Потомство отображается горизонтальным рядом карточек.
Источник данных:
- `pedigree.foals`
Каждый элемент массива отображается отдельной карточкой.
В карточке потомка отображаются:
1. Фото или плейсхолдер.
2. Кличка.
3. Иконка пола.
4. Пол + год рождения.
5. Масть.
6. Второй родитель.
7. Меню дополнительных действий.
Пример содержания карточки:
```
Stella ♀Кобыла · 2022ГнедаяОтецAtlas Thunder
```
или:
```
Orion ♂Жеребец · 2024ВоронойМатьAurora
```
---
## 9. Пустое состояние потомства
Если `pedigree.foals` пустой массив, вместо карточек показывать плейсхолдер.
Плейсхолдер располагается в центре блока потомства.
Текст:
```
Потомство отсутствуетУ этой лошади пока нет зарегистрированных потомков.
```
Кнопка “Добавить потомка” при этом остаётся в правой части заголовка блока.

---
## 10. Лошадь без фотографии
Если у любой лошади нет фотографии:
- текущая лошадь;
- отец;
- мать;
- потомок;
нужно показывать одинаковый плейсхолдер изображения.
Фото считается отсутствующим, если:
- `photos` отсутствует;
- `photos = []`;
- в массиве нет доступного `url`.
Карточка не должна ломаться и не должна менять структуру.
---
## 11. Меню дополнительных действий
Меню дополнительных действий отображается:
- у карточки отца;
- у карточки матери;
- у каждой карточки потомка.
Меню открывается по кнопке в правой части карточки.
### Пункты меню
Для существующей связи меню содержит:
```
Удалить связь Заменить связь Перейти
```
### 11.1. Удалить связь
Удаляет связь между текущей лошадью и выбранной карточкой.
Для отца:
- удаляется связь `pedigree.sire`.
Для матери:
- удаляется связь `pedigree.dam`.
Для потомка:
- удаляется родительская связь между текущей лошадью и выбранным потомком.
После удаления карточка родителя должна перейти в состояние “Не указан / Не указана”.
После удаления связи с потомком карточка потомка исчезает из списка потомства.
Если после удаления потомков не осталось, показывается пустое состояние потомства.

---

### 11.2. Заменить связь
Открывает сценарий выбора другой лошади.
Для отца:
```
Заменить отца
```
Для матери:
```
Заменить мать
```
Для потомка:
```
Заменить потомка
```
После выбора новой лошади связь обновляется, карточка перерисовывается.

---

### 11.3. Перейти

Открывает карточку выбранной лошади.

Для перехода использовать:

- `slug`, если он есть;
- иначе `id`.

Пункт “Перейти” не показывать, если связь отсутствует.

---
## 12. Нижняя панель действий
Внизу модального окна всегда отображается панель действий.
Справа располагаются кнопки:
```
Закрыть
```

### Закрыть
Закрывает модальное окно

---
## 13. Состояния загрузки
При открытии модального окна:
- если данные уже есть в DTO, показывать их сразу;
- если родословная догружается отдельно, показывать состояние загрузки внутри модального окна.
При загрузке отдельных действий:
- замена связи;
- удаление связи;
- добавление потомка;
- сохранение;
нужно блокировать повторное действие, чтобы пользователь не отправил запрос дважды.
---
## 14. Важные UX-правила
1. Не показывать отдельные кнопки “Найти отца”, “Найти мать”, “Создать новую лошадь” в основном состоянии модального окна.
2. Все действия над связями должны быть спрятаны в меню карточки.
3. У родителей и потомков должен быть одинаковый паттерн управления через меню.
4. Потомство не редактируется как ручной список. Оно отображает связи из данных.
5. Пустые состояния должны занимать место соответствующего блока, а не скрывать блок полностью.
6. Отсутствующий отец или мать должны отображаться как отдельная карточка-слот, чтобы пользователь понимал, что связь можно добавить.
7. В модальном окне показывается только одно поколение родителей.
8. Полное дерево поколений здесь не отображается.
9. Карточка текущей лошади находится в центре и не имеет меню действий.
10. Пользователь должен иметь возможность изменить родословную, не покидая модальное окно.
## 15. Логика сохранения
В подвале модального окна не должно быть кнопки сохранения. Все изменения применяются незамедлительно

# UX/UI модального окна связи родословной
## 1. Назначение
Модальное окно предназначено для:
- выбора существующей лошади;
- создания новой связи;
- замены существующей связи в родословной.
Окно открывается поверх модального окна родословной.
Используется в сценариях:
- добавить отца;
- заменить отца;
- добавить мать;
- заменить мать;
- добавить потомка;
- заменить потомка.
---
## 2. Источник данных
Для отображения списка используется тот же DTO лошади.
Используются поля:
- `id`
- `slug`
- `name`
- `sex`
- `breed`
- `coat_color`
- `bdate`
- `bdate_formatted`
- `bdate_mode`
- `photos`
- `this_stable`
---
## 3. Общая структура модального окна
Модальное окно состоит из:
1. Шапки.
2. Поисковой строки.
3. Блока результатов.
4. Нижней панели действий.
Структура:
```
[ Шапка ][ Поиск ][ Список найденных лошадей ][ Нижняя панель ]
```
---
## 4. Размер и поведение
Это вторичное модальное окно.
Оно открывается поверх модального окна родословной.
При открытии:
- фон затемняется;
- родительское модальное окно остаётся под ним;
- взаимодействие с родительским окном блокируется.
Модальное окно должно поддерживать:
- вертикальный скролл списка;
- фиксированную нижнюю панель действий;
- фиксированную поисковую строку.

можно дополнительно показывать индикатор:
```
В этой конюшне
```
Это необязательный UX-элемент, но полезный для ускорения выбора.

---
## 15. Выбор лошади
Справа карточки располагается радиокнопка выбора.
В один момент времени может быть выбрана только одна лошадь.

---
## 16. Выделение выбранной карточки
При выборе:
- активируется радиокнопка;
- карточка получает активное состояние.
Повторный клик по уже выбранной карточке не снимает выбор.

---
## 17. Поведение при клике
Клик по любой области карточки:
- выбирает карточку;
- активирует радиокнопку.

---
## 18. Скролл списка
Список результатов должен:
- иметь вертикальный скролл;
- не двигать нижнюю панель;
- не двигать поисковую строку.

---
## 19. Пустое состояние поиска
Если ничего не найдено:
показывать плейсхолдер.
Текст:
```
Ничего не найдено. Попробуйте изменить поисковый запрос
```

---

## 20. Состояние загрузки
Во время поиска:
- список блокируется;
- отображается состояние загрузки.
Во время сохранения:
- кнопка “Сохранить” блокируется;
- повторное сохранение невозможно.
---
## 21. Нижняя панель действий
Внизу модального окна располагается фиксированная панель действий.
Справа находятся кнопки:
```
Отмена Сохранить
```
### Кнопка “Отмена”
Закрывает модальное окно без изменений.
Выбранная лошадь не применяется.
### Кнопка “Сохранить”
Применяет выбранную лошадь к текущей связи.
### Поведение после сохранения
После успешного сохранения:
- вторичное модальное окно закрывается;
- пользователь возвращается в окно родословной;
- соответствующая карточка обновляется:
    - отец;
    - мать;
    - потомок.
---
## 27. Производительность
Поиск должен поддерживать:
- большое количество лошадей;
- пагинацию или lazy loading;
- виртуализацию списка при необходимости.
---
## 10. Важные UX-правила
1. Пользователь не должен вручную вводить ID лошади.
2. Основной сценарий — поиск и выбор из списка.
3. Карточки должны быть идентичны карточкам из модального окна родословной, но:
    - занимать 100% ширины;
    - быть рассчитаны на вертикический список.
4. Радиокнопка должна быть единственным способом подтверждения выбора внутри списка.
5. Выбор должен происходить по клику на всю карточку, а не только по радиокнопке.
6. Модальное окно не должно закрываться автоматически после выбора карточки — только после нажатия “Сохранить”.
7. Поиск должен работать без перезагрузки окна.
8. Если найден только один результат, он не должен выбираться автоматически.
9. Список результатов должен обновляться динамически во время ввода.
10. Поиск не должен сбрасывать текущий выбранный элемент, если он всё ещё присутствует в результатах.
