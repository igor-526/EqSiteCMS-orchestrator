# Архитектура и Сервисы

В данном документе описана высокоуровневая архитектура EqSimeCMS, роли микросервисов, инфраструктурных компонентов и способы их взаимодействия.

## 🚀 Общая архитектура

Проект построен по событийно-ориентированной микросервисной (и Service-Oriented) архитектуре.
Стек четко разделяет **Сбор/Аналитику данных**, **Бизнес-логику API** и **UI/Пользовательский интерфейс**.
Тяжёлые вычисления (кластеризация, сбор сторонней статистики) вынесены в асинхронные воркеры.

---

## 🏗 Инфраструктура (Databases / Brokers)

- **PostgreSQL**: Основная транзакционная БД. Хранит пользователей, проекты, балансы, настройки сущностей.
- **Redis**: In-memory хранилище для кэширования, хранения временных состояний и поддержки очередей для воркеров.

---

## ⚙️ Основные сервисы (Микросервисы)

### 1. Backend Core (`services/be`)

**Технологии:** Python, FastAPI, Clean Architecture (core, api, repositories), SQLAlchemy, Alembic.
**Роль:** Основной шлюз бизнес-логики (API) приложения.
**Функциональность:**

- **Обработка API запросов:** REST API для Фронтенда (кабинет, дашборды, цены, лошади, прочие данные).
- **Управление пользователями:** Регистрация, авторизация.

### 2. Frontend (`services/fe`)

**Технологии:** React, Next.js, Turbopack, TailwindCSS (или UI-kit). Feature-sliced design подход.
**Роль:** Клиентская часть (SSR/CSR SPA), полностью управляемая Backend-ом (Backend First).
**Функциональность:**

- **Страницы и Роутинг (**`src/app`**):** Содержит бизнес-логику UI-страниц (Дашборды, Цены, Лошади, Настройки).
- **Функциональные модули (**`src/features`**):** Изолированные фичи UI. Решения принимаются на основе данных из API.
- **Ограничения:** В `fe` запрещена тяжелая бизнес-логика. Он строго рендерит то, что агрегировано и обработано на Backend'е.
